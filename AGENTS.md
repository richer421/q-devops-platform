本文件是 `q-devops-platform` 子模块的 AI 开发约束（对 Codex/Agent 生效）。

## 目标

- 维护一个可持续演进的前端工程。
- 严格遵守既有技术栈版本，尤其是 Ant Design 版本。
- 通过组件化与页面化保持结构清晰。

## 开发硬约束

### 1) TS-only（必须）

- 所有新增代码必须使用 TypeScript：仅允许 `.ts` / `.tsx`。
- 禁止新增 `.js` / `.jsx` 业务代码。
- 保持 `tsconfig` 的 `strict: true`。

### 2) 严禁 `any`（必须）

- 禁止显式 `any`。
- 禁止通过断言链绕过类型系统（例如无理由的 `as any`）。
- 类型不明确时，优先使用 `unknown` + 类型守卫，而不是 `any`。

### 3) 组件化 + 页面化（必须）

- 页面（路由级）放在 `src/pages/*`，负责页面编排与数据装配。
- 组件放在 `src/components/*`，按业务域拆分，保证单一职责与复用性。
- 公共工具放在 `src/lib/*`，静态数据放在 `src/data/*`。
- 禁止把复杂业务逻辑直接堆在页面文件中。

### 4) 样式约束

- 统一使用 Ant Design 组件体系，不使用 Tailwind/Flowbite。
- 新功能和重构场景中，尽可能优先使用 Ant Design 官方组件与能力，并遵循 Ant Design 最佳实践。
- 禁止新增 `className` 与自定义 CSS 文件，样式通过 Antd 组件能力和内联 `style` 完成。
- 视觉风格需保持一致（页面背景、边框层级、按钮和状态色）。

### 5) 表格设计规范（必须）

- 表格优先采用 Ant Design Table，并基于统一交互与视觉规则实现。
- 对齐规则：
  - 文字、标题、描述、备注：左对齐。
  - 数字、金额、ID、序号：右对齐。
  - 状态、开关、操作按钮、图标：居中对齐。
- 列宽规则：
  - 短列（序号、状态、操作）：固定宽度，默认控制在 `80px` 到 `120px`。
  - 常规列（名称、标题）：使用最小宽度或自然宽度，并允许弹性拉伸。
  - 长列（备注、描述）：最大宽度控制在 `300px`，禁止撑破表格主体。
- 内容溢出规则：
  - 超长文本统一使用单行省略。
  - 省略内容必须支持 hover 后通过 Tooltip 查看全文。
- 视觉规则：
  - 只保留表格外框与横向分隔线，不绘制竖向分隔线。
  - 统一行高为 `48px`，保持清爽、稳定的阅读节奏。
- 适配规则：
  - 表格外层必须支持横向滚动。
  - 禁止通过压缩列宽或强行换行破坏信息层次。
- 实现要求：
  - 新表格组件优先抽象公共列策略与溢出策略，避免每个页面重复定义一套风格。
  - 若业务有特殊列样式，必须在不破坏上述基础规范的前提下做局部扩展。

### 6) 调整与回退流程（必须）

- 当用户明确指出本轮新增改动不符合预期，并要求继续调整时，第一步应优先回退本轮偏离预期的改动。
- 不允许在错误方向的实现上继续叠加修补，避免引入更多视觉噪音、状态污染和判断偏差。
- 回退后，应基于最近一个被用户认可的稳定状态重新实施调整。
- 如仅有局部实现错误，允许精确回退局部改动；但前提是能够明确界定错误边界，不能模糊处理。

### 7) 前端测试方式（必须）

- 前端测试与验收默认基于 Playwright MCP 执行真实页面测试。
- 涉及业务流程、页面交互、联调链路时，优先通过 Playwright MCP 覆盖关键步骤，而不是补充脆弱的 DOM 结构测试。
- 测试前应先确保本地依赖、开发服务与相关后端代理已经就绪。
- 完成验收后，需要记录关键步骤、结果以及必要的截图或报告路径。

## 交付前检查

在子模块根目录执行：

```bash
make lint
make build
```

前端页面测试不再以仓库内单元测试为默认手段；如需页面验收，使用 Playwright MCP 执行。

出现失败时，不得宣称任务完成。
