# q-devops-platform 开发指南

本文件用于约束本前端模块的技术栈与开发方式。

## 技术栈（锁定）

当前项目基于以下版本开发：

- Node.js: `>=22.0.0`
- pnpm: `10.30.3`
- React / React DOM: `18.3.1`
- react-router / react-router-dom: `7.13.1`
- Ant Design: `5.29.3`
- Vite: `6.3.5`
- @vitejs/plugin-react: `4.7.0`
- TypeScript: `5.8.2`
- ESLint / typescript-eslint: `9.22.0` / `8.26.1`
- Prettier: `3.5.3`

## 版本管理要求

1. 默认不修改技术栈版本。
2. 如必须升级依赖，需在变更中明确原因、风险与回滚方案。
3. 涉及版本改动时，必须同步更新 `package.json`、`pnpm-lock.yaml` 以及本文档与 `AGENTS.md`。
4. Ant Design 版本变更需要明确审批，并同步更新文档。

## 开发要求

### 1. 组件化与页面化

- 页面层：`src/pages/*`，负责路由页面、数据组装、页面级状态。
- 组件层：`src/components/*`，负责可复用 UI 与业务组件。
- 工具层：`src/lib/*`，放通用函数与类型相关工具。
- 数据层：`src/data/*`，放 mock 或静态结构化数据。

要求：页面只做编排，不堆积可复用逻辑；可复用逻辑应下沉为组件或工具。

### 2. TypeScript 强制

- 新增与修改代码必须是 TypeScript（`.ts` / `.tsx`）。
- 禁止提交 `.js` / `.jsx` 业务代码。
- 维持严格类型模式（`strict: true`）。

### 3. 严禁 `any`

- 禁止显式 `any` 与 `as any`。
- 遇到不确定类型，使用 `unknown`、联合类型、泛型、类型守卫解决。

### 4. 样式规范

- 使用 Ant Design 组件为主。
- 新功能和重构场景中，尽可能优先使用 Ant Design 官方组件与能力，并遵循 Ant Design 最佳实践。
- 禁止新增 Tailwind/Flowbite 依赖与用法。
- 禁止在 TSX 中使用 `className`，样式通过 Antd + `style` 完成。
- 禁止新增自定义 CSS 文件。

### 5. 表格设计规范

- 表格优先使用 Ant Design Table，并遵循统一设计规则。
- 对齐规则：
  - 文字、标题、描述、备注：左对齐。
  - 数字、金额、ID、序号：右对齐。
  - 状态、开关、操作按钮、图标：居中对齐。
- 列宽规则：
  - 短列（序号、状态、操作）：固定宽度，默认在 `80px` 到 `120px`。
  - 常规列（名称、标题）：使用最小宽度或自然宽度，并允许弹性拉伸。
  - 长列（备注、描述）：最大宽度控制在 `300px`，避免撑破表格。
- 内容溢出：
  - 超长文本统一使用单行省略号。
  - 省略内容必须支持 hover Tooltip 查看全文。
- 视觉样式：
  - 仅保留表格外框与横向分隔线，不绘制竖线。
  - 行高统一为 `48px`，保持信息密度稳定。
- 适配规则：
  - 表格容器必须支持横向滚动。
  - 不允许通过强行压缩列宽或文本换行来破坏布局。
- 工程要求：
  - 优先抽象公共表格能力，例如列对齐策略、溢出展示、固定宽度策略。
  - 特殊业务表格只能在统一规范上做局部扩展，不能各写一套视觉规则。

### 6. 调整与回退流程

- 当用户明确指出本轮新增改动不符合预期，并要求继续调整时，第一步应优先回退本轮偏离预期的改动。
- 不允许在错误方向的实现上继续叠加修补，避免引入更多视觉噪音、状态污染和判断偏差。
- 回退后，应基于最近一个被用户认可的稳定状态重新实施调整。
- 如果只是局部实现错误，可以精确回退局部改动；前提是错误边界足够明确，不能含混处理。

### 7. 前端测试方式

- 前端测试与页面验收默认使用 Playwright MCP。
- 只要涉及业务流程、交互链路、真实联调，就优先执行 Playwright MCP 的真实页面测试。
- 不再默认补充仓库内前端单元测试；只有用户明确要求时，才额外设计其他测试形式。
- 验收完成后，应反馈关键步骤、结果与必要的截图或报告路径。
- 只要本轮修改影响了页面交互、表单提交流程、筛选切换、列表刷新、分页滚动、弹窗打开关闭、数据详情展示等可见行为，Playwright MCP 验收就是强制门禁，不是建议项。
- 没有 Playwright MCP 实际执行证据时，不允许宣称“前端已完成”“可提交”“可合并”，也不能用 `make lint` / `make build` 代替页面验收。

## 质量门禁

每次提交前在模块目录执行：

```bash
make lint
make build
```

若改动涉及交互链路，还必须补充一次真实 Playwright MCP 验收，输出以下信息：

1. 启动环境与前置条件。
2. 验收步骤。
3. 验收结果。
4. 截图或报告路径。

前端测试默认通过 Playwright MCP 完成，不以仓库内测试脚本作为默认要求。

任一命令失败，则该变更不应提交。
